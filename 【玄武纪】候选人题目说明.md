# 合同 AI 审查与知识库检索笔试说明

版本：2026-05-26  
完成时间：收到题目后 48 小时内  
材料类型：PDF 测试材料（扫描件），内容为合成企业文档，不包含真实客户信息  

## 1. 任务背景

玄武纪人工智能科技有限公司正在面向存量的几千家国内大中型企业、国央企、上市公司等客群，提供基于合同审查、履约管理、AI 问答等场景的大模型能力输出。本次笔试提供一份综合测试 PDF 文档，文档内容模拟“智能印章与合同审查平台建设项目”的合同正文、项目 SOW、报价表、验收方案、运维条款、数据安全条款、流程图和附件说明。

你需要基于这份文档完成两个方向的结果：

1. RAG 问答：把文档处理成可检索、可引用、可追问的知识库，并回答指定问题。
2. 合同审查：识别合同和附件中的潜在风险，输出结构化审查结果。

本题重点不是做一个漂亮页面，而是判断你是否能处理复杂企业文档，并把结果做成可复现、可回溯、可评测的工程输出。

## 2. 提供材料

- `【玄武纪】综合测试文档.pdf`

## 4. RAG 问答任务

请基于同一份文档完成 3 类问题。每个回答都必须给出引用来源，引用应能定位到章节、表格、图示或附件段落。

### Q1：简单事实问题

问题：

> 本项目包含哪些主要系统模块？请列出模块名称，并说明每个模块的主要用途。

测试目的：

- 是否能定位单点事实。
- 是否能从“系统功能范围”“交付物”“流程图”等位置提取一致答案。
- 是否能给出清晰引用。

### Q2：多轮问答

请按顺序回答以下三轮问题。后两轮必须理解前文上下文，不能把每一问当成孤立问题处理。

第一轮：

> 本项目需要交付哪些主要成果物？

第二轮追问：

> 这些成果物分别对应哪些验收材料？

第三轮追问：

> 如果验收材料缺失，可能影响哪些付款节点？请说明依据。

测试目的：

- 是否能做多轮问题改写。
- 是否能维护对“这些成果物”的上下文指代。
- 是否能把交付物、验收材料和付款节点关联起来。

### Q3：全文复杂推理问题

问题：

> 请综合判断本项目的付款安排、验收条件、交付计划和附件说明之间是否存在冲突或不一致。请逐条说明依据，并指出哪些问题需要人工复核。

测试目的：

- 是否能跨合同正文、SOW、报价表、验收章节、流程图和附件说明检索。
- 是否能基于多处细节做综合判断。
- 是否能区分“明确事实”“合理推断”和“需要人工确认”。

## 5. `qa_results.json` 输出格式

请输出合法 JSON。示例结构如下，可增加字段，但不要删除核心字段。

```json
[
  {
    "question_id": "Q1",
    "question_type": "simple",
    "question": "本项目包含哪些主要系统模块？请列出模块名称，并说明每个模块的主要用途。",
    "answer": "你的回答",
    "citations": [
      {
        "source_id": "doc-section-or-block-id",
        "section": "3.2 系统功能范围",
        "quote": "原文短引用",
        "reason": "该引用如何支持回答"
      }
    ],
    "retrieval_notes": {
      "chunk_strategy": "例如：按标题层级切分，表格独立 chunk，图示转文本描述",
      "metadata_fields": ["section_path", "page_hint", "block_type"],
      "retrieval_method": "例如：embedding top_k=8 + BM25 + rerank",
      "multi_turn_handling": "如不涉及多轮可写 null"
    },
    "confidence": 0.0
  }
]
```

要求：

- `citations` 不少于 1 条，复杂推理问题建议不少于 3 条。
- `quote` 只放必要原文，不要整段复制。
- `retrieval_notes` 必须写真实做法，不能只写“使用 RAG”。
- `confidence` 取值 0 到 1。

## 6. 合同审查任务

请基于同一份文档输出潜在风险。至少识别 6 条风险，鼓励识别更多，但不要为了凑数输出没有证据的风险。

审查重点包括：

- 主体一致性。
- 金额一致性。
- 付款条件与验收/交付条件是否匹配。
- 交付计划是否前后一致。
- 验收标准是否明确。
- 附件是否完整。
- 违约责任是否明显不对等。
- 数据安全、私有化部署和第三方接口是否存在冲突。
- 流程图中的人工复核责任是否在正文中落实。
- 扫描/OCR/表格解析不确定项是否进入人工复核。

## 7. `review_results.json` 输出格式

请输出合法 JSON。示例结构如下，可增加字段，但不要删除核心字段。

```json
[
  {
    "risk_id": "R001",
    "risk_type": "金额一致性",
    "severity": "high",
    "title": "合同金额与报价表合计金额不一致",
    "evidence": [
      {
        "source_id": "doc-section-or-block-id",
        "section": "1.3 合同金额",
        "quote": "原文短引用"
      }
    ],
    "reason": "说明为什么构成风险",
    "suggestion": "给出修改建议或复核建议",
    "needs_human_review": true,
    "confidence": 0.0
  }
]
```

要求：

- `severity` 只能使用 `low`、`medium`、`high`。
- 每条风险必须有原文证据。
- 如果证据来自表格、图示或附件，请在 `section` 中写清楚。
- 如果只是模型推断，必须标记 `needs_human_review: true`。

## 8. 分块和检索策略说明

请在 `chunking_and_retrieval.md` 中说明以下内容：

1. 文档解析方式：如何处理 PDF 扫描件、OCR 文本、表格、图示、页码标记、附件和签署页。
2. 分块策略：按标题层级、段落、固定长度、递归切分、表格独立切分，还是混合方式。
3. chunk metadata：至少说明是否包含 `section_path`、`block_type`、`page_hint`、`table_id`、`source_text` 等字段。
4. 检索策略：embedding 模型、BM25、hybrid search、rerank、top_k、过滤条件。
5. 多轮问答策略：如何处理追问、上下文压缩、问题改写和引用延续。
6. 长文档策略：如果文档扩展到 200 页，你会如何避免遗漏和前后矛盾。
7. 表格和图示策略：如何处理 PDF 表格、跨页表格、流程图和架构图。
8. 失败边界：哪些情况会低置信度或进入人工复核。

必须提交源码。我们会重点看：

- 文档解析和分块是否真实实现。
- 检索和审查结果是否可复现。
- 是否只是一次性把全文塞给大模型。
- 是否有清晰的数据结构和中间结果。

不强制要求：

- 不强制做前端。

- 不强制做登录、权限、数据库管理后台。

允许使用：

- Python、Java、JavaScript/TypeScript 等语言。
- LangChain、LlamaIndex、Dify、RAGFlow、自研脚本等。
- OCR、视觉模型、Embedding、Rerank、向量库、关键词检索。
- 商业模型 API 或本地模型。

如果使用商业 API，请不要提交真实 key。

## 10. 评分重点

| 评分项 | 分值 | 重点 |
|---|---:|---|
| 文档解析与结构化 | 20 | 标题、段落、表格、图示、附件、签署页是否可定位 |
| 分块与检索策略 | 20 | 是否有合理 chunk、metadata、hybrid/rerank、多轮处理 |
