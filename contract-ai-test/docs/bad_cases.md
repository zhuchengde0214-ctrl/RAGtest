# Bad Cases — 失败案例分析

---

## Bad Case 1: 表格跨页导致金额合计错误

- **问题**: PDF 报价表跨第 5-6 页，表头在第 5 页，数据行延续到第 6 页。OCR 提取第 6 页表格时缺少表头信息，导致金额列与实际项目对应关系出错，合计金额计算结果与实际不符。
- **预期**: 应正确识别跨页表格，合并表头和数据行，金额合计应等于正文合同金额。
- **实际**: 第 6 页的表格被识别为独立表格，缺少列名映射，金额列无法关联到具体收费项目。
- **原因**:
  1. 逐页 OCR 缺乏跨页上下文感知
  2. 表格结构恢复未考虑跨页场景
  3. 缺少表头继承逻辑
- **改进**:
  1. 实现跨页表格检测：相邻页连续出现 `[TABLE_*]` 标记时合并处理
  2. 表头继承：跨页表格自动继承前页表头
  3. 金额校验：提取所有金额数字，与合同总金额交叉验证，偏差 > 1% 时标记人工复核
  4. 可使用专门的表格提取模型（如 Table Transformer）

## Bad Case 2: 多轮问答中指代消解失败导致检索偏离

- **问题**: Q2 第二问"这些成果物分别对应哪些验收材料？"中，"这些成果物"的指代消解不完整。改写后的 query 未包含第一轮问答中列出的具体交付物列表，导致检索偏向验收材料的一般性描述，而非每个交付物对应的具体验收材料。
- **预期**: 问题改写应将"这些成果物"替换为第一轮中识别的具体交付物名称，如"智能印章管理平台、合同审查系统、移动审批 APP..."，然后分别检索各交付物的验收材料。
- **实际**: 改写后的问题为"本项目需要交付的主要成果物分别对应哪些验收材料？"，虽然语义正确但缺少具体交付物关键词语义锚点，BM25 检索效果不佳。
- **原因**:
  1. 问题改写 prompt 未要求保留前轮的具体实体信息
  2. 改写过度抽象化，丢失了可用于检索的关键词
- **改进**:
  1. 问题改写时保留前轮回答中的关键实体和术语
  2. 增加 entity extraction 步骤，将提取的实体直接追加到改写后的 query 中
  3. 使用 HyDE (Hypothetical Document Embeddings) 生成假设性文档来补充检索

## Bad Case 3: 扫描件水印/印章导致文本提取噪声

- **问题**: PDF 签署页包含公司印章和"内部资料"水印，Claude Vision API 在提取文本时，将水印文字"内部资料"误识别为正文内容，混入签署页的区块中。同时，印章导致部分签名文字被遮挡，提取的签名信息不完整。
- **预期**: 应区分水印文字与正文内容，水印应标记为 artifact 而非正文；印章区域的文字应标记为低置信度。
- **实际**: "内部资料"被作为普通文字提取到签署页段落中；印章覆盖区域的文字缺失。
- **原因**:
  1. Vision prompt 未明确要求忽略水印
  2. 印章遮挡导致 OCR 无法读取底层文字
  3. PDF 渲染时未做去水印预处理
- **改进**:
  1. Vision prompt 增加"忽略水印、页码、页眉页脚等非正文内容"的指令
  2. 图像预处理：使用 OpenCV 检测印章颜色区域，尝试分离印章层和文字层
  3. 签署页特殊处理：识别 `[SIGNATURE]` 标记的页面区域，降低自动提取置信度，优先进入人工复核
  4. 多模型交叉验证：同一页面用不同模型各跑一次，对比结果

## Bad Case 4: 嵌入模型对中文法律术语的语义理解不足

- **问题**: 检索"违约责任是否明显不对等"时，向量检索的 top 结果多为包含"违约"字样的段落，但遗漏了包含"赔偿上限"、"责任限制"、"免责条款"等段落。这些段落虽然没有直接出现"违约责任"四个字，但同样是判断责任对等性的关键内容。
- **预期**: 检索应覆盖"违约"、"赔偿"、"责任限制"、"免责"、"罚则"等语义相关但文字不同的内容。
- **实际**: OpenAI embedding 模型对中文法律术语的同义/近义理解有限，依赖字面匹配。BM25 同样受限于关键词匹配。
- **原因**:
  1. 通用 embedding 模型未针对中文法律领域微调
  2. 法律文本中存在大量同义词和近义表达
  3. 单一查询的语义覆盖范围有限
- **改进**:
  1. 查询扩展：对原始 query 生成法律术语的同义表达，多 query 并行检索后合并
  2. 使用针对中文优化的 embedding 模型（如 bge-large-zh-v1.5）
  3. 建立法律术语同义词表，在检索前自动扩展 query
  4. 对法律领域的关键概念做 query-to-concept 映射

## Bad Case 5: 长上下文下 LLM 遗漏关键证据

- **问题**: Q3 复杂推理时，LLM 需要同时分析付款安排、验收条件、交付计划三个方面的文档片段。但回答中遗漏了附件说明中关于"初验通过后支付 30%"的条款与正文"初验后支付 50%"的矛盾，只关注了正文内容而忽略了附件。
- **预期**: 应识别出正文与附件中同一付款节点的比例差异，并标记为金额/比例冲突风险。
- **实际**: LLM 注意力集中在前几个检索片段，对靠后的附件相关内容关注不足，导致了"Lost in the Middle"现象。
- **原因**:
  1. LLM 对长上下文中间部分信息关注度下降（Lost in the Middle）
  2. 检索排序将附件内容排在较后位置
  3. 未对矛盾性内容做专门的对比提示
- **改进**:
  1. 对检索结果按主题分组，每组分别让 LLM 分析后再汇总
  2. 使用 chain-of-thought 分步推理：先分析付款，再分析验收，再分析交付，最后综合比较
  3. 增加矛盾检测步骤：专门让 LLM 比较不同章节中涉及同一主题（如金额、时间、比例）的内容
  4. 对关键数值信息（金额、比例、日期）做结构化提取后再对比
