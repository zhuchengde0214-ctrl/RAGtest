# Bad Cases — 失败案例与改进

下列 5 个 case 都是在本工程上实际遇到 / 经过推演认定**仍可能再次发生**的失败模式。每条都对应了已落地的缓解措施和后续可继续优化的方向。

---

## Bad Case 1：跨页表格表头丢失导致金额对应错误

- **问题**：报价表横跨第 5-6 页，表头只在第 5 页。逐页 OCR 时第 6 页的数据行没有列名映射，"小计"列与项目名错位。
- **预期**：合并跨页表格，所有数据行都能正确归到列；表格合计金额能与正文合同总金额对账。
- **实际触发**：当 LLM 在第 6 页幻想出一个不一致的表头（如把"小计"输出为"合计"），合并去重逻辑判定表头不一致，跨页合并失败 → 退化为两个独立 table chunk。
- **原因**：
  1. Vision API 对续表表头的输出不稳定
  2. 合并判据严格按字符串相等
- **当前已做的缓解**：
  - `pdf_parser._merge_cross_page_tables` 优先合并；首行表头一致 / 后续无表头（直接数据行）两种情况都接受
  - 大表格 chunk 时显式保留 `header + separator` 行作为前缀，让每个 table chunk 独立可读
  - "金额一致性"审查维度专门检索 "合同总金额 / 报价合计"，一旦数字对不上即提示风险
- **后续改进**：
  1. 加金额结构化提取层：从 chunk 中正则抽出所有 `[\d,.]+\s*(元|万|％)`，跨章节做集合对账，差异 > 1% 自动标 `severity=high`
  2. 表格用专门的视觉模型（Table Transformer / Surya）替代单 prompt OCR
  3. 跨页表格判据从字符串相等改为 fuzzy 匹配（去空格、去标点、容许 1-2 字符 Levenshtein 距离）

## Bad Case 2：多轮问答指代消解过度抽象，丢失检索锚点

- **问题**：Q2 第 2 问"这些成果物分别对应哪些验收材料？" 改写时如果只把"这些成果物"换成"上述成果物"，BM25 召回会偏向"成果物"这种泛化词，错过具体模块名。
- **预期**：改写后必须保留第 1 轮答案中识别出来的具体名称（如"智能印章管理平台 / 合同审查系统 / 移动审批 APP / 运维监控平台"），让检索锚点足够具体。
- **实际**：若改写得过于"漂亮"，容易变成"本项目主要成果物分别对应的验收材料是什么"，BM25 几乎打不到任何具体 chunk。
- **当前已做的缓解**：`retriever.rewrite_query` 的 prompt 显式写明"保留前轮已识别出的关键实体名称（如系统模块名、合同条款编号、金额等），便于检索"；而且会把最近 6 条历史的 user / assistant 全部喂进去，让 LLM 看到具体名词。
- **后续改进**：
  1. 在改写前先从历史中正则抽取"专有名词候选集"（连续 4 字以上中文 + 包含数字的字段）作为 hint
  2. 同时跑 2 路检索：原 query + 改写 query，结果用 RRF 再融合
  3. HyDE — 让 LLM 基于历史先写一段"假设答案"做 embedding 检索

## Bad Case 3：印章/水印噪声混入正文

- **问题**：签署页有公司印章和"内部资料"水印；Vision 若未明确指示忽略，可能将"内部资料"误识别为正文段落，污染该页 chunk。
- **预期**：水印不进入正文；印章覆盖区的字段标记为低置信度。
- **当前已做的缓解**：
  1. Vision prompt 明确要求"忽略页眉页脚水印（如『内部资料』『第X页/共Y页』）"
  2. 印章遮挡的字段要求输出 `[?]` 占位
  3. `[?]` 占位会让 block 自动 `needs_review=True`
  4. 签署区的所有 needs_review block 会被审查阶段的"OCR 不确定项"维度聚合成一条 medium 级风险
- **失败边界**：水印颜色与正文颜色相近、或水印密集覆盖正文 → Vision 仍可能误识别。此时会有重复 chunk 或语义混乱，需要人工核对原件。
- **后续改进**：
  1. 图像预处理：用 OpenCV 检测大面积红/绿/蓝色块（典型水印/印章颜色），生成 mask 后再喂 Vision
  2. 同一页跑 2 次 OCR（不同温度），若两次内容差异 > 阈值则标 needs_review

## Bad Case 4：通用 embedding 对中文法律近义词召回不足

- **问题**：检索"违约责任不对等"时，dense embedding 容易只召回字面含"违约"的段落，漏掉"赔偿上限/责任限制/免责条款"等近义表达。
- **当前已做的缓解**：
  1. 不靠 dense embedding 单打独斗：BM25 + dense 走 RRF 融合
  2. BM25 用 `jieba.cut_for_search` + 2-gram 兜底，"赔偿"、"违约金"、"罚则" 都会被切成可命中的 token
  3. 审查阶段的"违约责任对等性"维度配了 3 条候选 query：`违约责任 违约金 赔偿 责任限制 上限` / `免责 不可抗力 终止 解除` / `保密 知识产权 侵权`，多 query 召回后合并去重
  4. 每个维度独立调 LLM，避免 Lost-in-the-Middle
- **后续改进**：
  1. 把通用 embedding 换成 BGE/M3E 等中文优化模型（`bge-large-zh-v1.5`）
  2. 维护"法律术语同义词表"（违约金 ↔ 罚则 ↔ 赔偿金；初验 ↔ 阶段验收 ↔ 中期验收），检索前对 query 做扩展
  3. 用 LLM 先生成 5 条不同表达的等价 query，并行检索后合并

## Bad Case 5：Q3 复杂推理在长上下文中遗漏附件证据（Lost in the Middle）

- **问题**：Q3 要求跨章节判断付款 / 验收 / 交付 / 附件之间的一致性。如果一次性把 12+ 个 chunk 都喂给 LLM，靠后位置的附件 chunk 容易被忽略，导致只发现正文内的矛盾，漏掉正文-附件之间的矛盾。
- **当前已做的缓解**：
  1. **子问题分解**：Q3 先用 LLM 拆成 3-5 个独立子问题，每个子问题各自检索
  2. **多路结果合并**：按 chunk_id 合并去重，多个子问题命中的 chunk 分数累加（隐式重要性提升）
  3. **结构化输出**：要求 LLM 直接输出 `conflicts[]` 数组，每条带 `topic / conclusion_class / evidence`，避免自由发挥
  4. **conclusion_class 三分类**：fact / inference / human_review；非 fact 自动设 `needs_human_review=true`
  5. **审查阶段独立兜底**：合同审查的"付款 vs 验收/交付匹配""附件完整性""交付计划一致性"三个维度独立跑，不依赖 Q3 的结果
- **后续改进**：
  1. Map-Reduce：每个子问题先单独让 LLM 总结 3-5 句事实，再把多份事实喂给汇总 LLM 做对比，避免长上下文
  2. 数值字段先做结构化抽取（金额、比例、时间），用代码做集合对账，把对账结果作为 hint 喂给 LLM
  3. 对每个"待对比对"显式构造 prompt（"以下两段文字是否有矛盾？"），让 LLM 二选一回答

## 附：每个 case 的复现路径

| Case | 怎么复现 |
|---|---|
| 1 | `python3 src/main.py --max-pages 6` 后查看 `outputs/chunks.json` 中 `block_type=table` chunk 的 `pages` 字段 |
| 2 | 删除 `outputs/qa_results.json` 重跑，对比 Q2-2 的 `rewritten_question` 字段 |
| 3 | 检查 `outputs/parsed_document.json` 中 `block_type=signature` 与 `needs_review=true` 的 block |
| 4 | 在 retriever 上用 `search('赔偿上限', top_k=5)` vs `search('违约责任', top_k=5)` 对比命中差异 |
| 5 | 检查 `outputs/qa_results.json` 中 Q3 的 `conflicts[].evidence`，看是否有 `pages` 涉及附件页 |
